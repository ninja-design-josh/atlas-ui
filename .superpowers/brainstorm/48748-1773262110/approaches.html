Three Approaches to World-Class Atlas UI

Which strategy best matches NinjaCat's goals? Click to select.

A

Deep Foundation First

Don't add components yet. Build the infrastructure that makes AI tools work perfectly with the 9 existing components — then expand systematically.

What you build

  • Component registry (JSON/MDX) — machine-readable spec AI can consume
  • llms.txt + AI context files for Claude, Replit, v0
  • Figma Code Connect — two-way sync between Figma & components
  • Interactive docs upgrade (live props editor, dark/light toggle)
  • Design token automation (Figma → CSS → code pipeline)
  • Accessibility audit + ARIA on all 9 components

Trade-offs

  • Slower to feel "complete" — still 9 components for a while
  • High upfront infrastructure investment
  • Payoff is compounding — each new component benefits from the system

Best if: You want AI tools working consistently before expanding breadth

B

Storybook-Powered Platform

Migrate to Storybook as the documentation backbone — the industry standard used by MUI, Radix, and every major design system. AI tools already understand Storybook conventions.

What you get

  • Interactive controls panel (live prop editing out of the box)
  • Figma plugin built into Storybook (side-by-side design vs. code)
  • Visual regression testing via Chromatic
  • Accessibility addon (a11y checks per component)
  • Industry-standard structure AI tools know deeply
  • Auto-generated docs from TypeScript types

Trade-offs

  • Migration cost — rewrite all stories for existing 9 components
  • Storybook has its own opinions, may conflict with Next.js App Router
  • Heavier toolchain to maintain

Best if: You want the most battle-tested foundation with designer tooling built in

C

Expand Breadth + AI Context in Parallel

Rapidly grow to 40–60 components matching Figma coverage, while building AI-friendly structure (consistent APIs, JSDoc, TypeScript) as you go. Custom docs site gets upgraded alongside.

What you build

  • Full component suite: modals, tables, toasts, tooltips, menus, tabs, accordions, datepickers, etc.
  • Consistent prop naming conventions across all components
  • JSDoc on every prop and component
  • Upgraded custom docs site with search, filtering, interactive previews
  • Component registry JSON alongside each component file

Trade-offs

  • Risk of inconsistency if moving fast without strong conventions
  • Docs site needs significant investment to match Storybook's power
  • Harder to maintain — no automated visual regression

Best if: You need more component coverage immediately for AI tools to be useful

My recommendation

A hybrid of A + C: build the AI infrastructure foundation (registry, llms.txt, Figma Code Connect) in parallel with expanding component breadth — but skip the Storybook migration to avoid rewrite cost. The custom docs site can evolve into something more interactive without the toolchain overhead.
